# Adding a New Versioned Spec to spec.gesslar.dev

This site uses Docusaurus with independent `@docusaurus/plugin-content-docs`
instances for each specification. Each spec has its own route, sidebar,
versioning, and homepage card. Only versioned docs are served
(`includeCurrentVersion: false`).

## Information Needed

Before scaffolding, gather from the user:

1. **Spec name** (`{id}`) — lowercase identifier for paths/routes/plugin ID
   (e.g., `lpml`, `lpcdoc`). No spaces.
2. **Display title** (`{title}`) — human-readable name for the homepage card
   and index page (e.g., "LPML", "LPCDoc").
3. **One-line description** (`{description}`) — shown on the homepage card.
4. **Initial version** (`{version}`) — e.g., `1.0.0`.
5. **Codicon icon** (`{icon}`) — VS Code codicon class for the homepage card
   (e.g., `codicon-file-code`, `codicon-book`). Full list:
   https://microsoft.github.io/vscode-codicons/dist/codicon.html
6. **Sidebar style** — `autogenerated` (default) or `manual` (explicit items).

`{Id}` below means the spec name with first letter capitalized (e.g., `Lpml`).

## Files to Create

### 1. `specs/{id}/index.md`

```markdown
---
title: {title} Specification
sidebar_position: 1
---

# {title} Specification

{description}

## Version

Version: {version}
Created: {today's date, YYYY-MM-DD}
```

### 2. `sidebars{Id}.js`

For autogenerated:

```js
const sidebars = {
  {id}Sidebar: [{type: 'autogenerated', dirName: '.'}],
}

export default sidebars
```

For manual, use an explicit items array containing `"index"` to start.

### 3. `{id}_versions.json`

```json
["{version}"]
```

### 4. `{id}_versioned_docs/version-{version}/`

Create this directory and copy `specs/{id}/index.md` into it.

### 5. `{id}_versioned_sidebars/version-{version}-sidebars.json`

```json
{
  "{id}Sidebar": [
    {
      "type": "autogenerated",
      "dirName": "."
    }
  ]
}
```

(Or equivalent manual structure.)

## Files to Edit

### 6. `docusaurus.config.js`

Add to the `plugins` array:

```js
[
  '@docusaurus/plugin-content-docs',
  {
    id: '{id}',
    path: 'specs/{id}',
    routeBasePath: '{id}',
    sidebarPath: './sidebars{Id}.js',
    includeCurrentVersion: false,
  },
],
```

Add to `themeConfig.navbar.items` (before the GitHub link):

```js
{
  type: 'docsVersionDropdown',
  docsPluginId: '{id}',
  position: 'left',
  className: 'version-dropdown version-dropdown--{id}',
},
```

### 7. `src/css/custom.css`

Add a version dropdown visibility rule following the existing pattern. Insert
it after the last `version-dropdown--` rule:

```css
html:has([class*='docMainContainer'] [href*='/{id}/']) .version-dropdown--{id} {
  display: inline-flex !important;
}
```

### 8. `src/pages/index.js`

Add an entry to the `specs` array:

```js
{
  title: '{title}',
  description: '{description}',
  href: '/{id}',
  icon: '{icon}',
},
```

## After Scaffolding

1. Run `npm run build` to verify the site builds without errors.
2. If it succeeds, tell the user the new spec is ready at `/{id}`.
3. Remind them that `specs/{id}/` is the working directory for content, and
   they must copy files to `{id}_versioned_docs/version-X/` for them to
   appear on the live site (since `includeCurrentVersion: false`).
